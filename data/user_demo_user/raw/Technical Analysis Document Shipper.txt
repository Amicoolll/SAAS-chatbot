bd.pickupPhoneNumber, bd.pickupAddressId, bd.pickupAddressCd, bd.pickupAddressType, bd.pickupApartment,bd.pickupStreetName, bd.pickupLandmark, bd.pickupLocality, bd.pickupCity, bd.pickupState, bd.pickupCountry, bd.pickupZipCode, bd.pickupNotes, bd.deliverBranchId, bd.deliverStartTimeWindow, bd.deliverEndTimeWindow, bd.deliverServiceTime, bd.deliverCustomerId, bd.deliverAccountCode, bd.deliverAccountName, bd.deliverEmail, bd.deliverPhoneNumber, bd.deliverAddressId, bd.deliverAddressCd, bd.deliverAddressType, bd.deliverApartment, bd.deliverStreetName, bd.deliverLandmark,sod.orderId, bd.deliverLocality, bd.deliverCity, bd.deliverState, bd.deliverCountry, bd.deliverZipCode, bd.bookingStatus, bd.deliverNotes, bd.deliveryType, bd.serviceType, bd.paymentType, bd.packagingType, bd.totalWeight, bd.totalValue, bd.totalNoOfItems, bd.basicShippingCost, bd.packagingCost, bd.totalShippingCost, bd.pickupAddress, bd.deliverAddress, sod.orderStatusCd, cms.name, bd.bookingType, bd.totalVolume, sod.orderState, sd.isPartialDeliveredFl, bad.returnBranchId, bad.returnStartTimeWindow,bad.returnEndTimeWindow, bad.returnServiceTime, bad.returnCustomerId, bad.returnAccountCode, bad.returnAccountName, bad.returnEmail, bad.returnPhoneNumber, bad.returnAddressId, bad.returnAddressCd, bad.returnAddresstype, pcbm.name AS pickupBranchName, dcbm.name AS deliverBranchName, bad.returnApartment, bad.returnStreetName, bad.returnLandmark, bad.returnLocality, bad.returnCity, bad.returnState, bad.returnCountry, bad.returnZipCode, bd.isReturnAllowedFl 
FROM bookingdetails bd 
INNER JOIN clientmaster cms ON bd.subClientId = cms.clientId 
LEFT JOIN bookingshipmentmapping bsm ON bd.bookingDetailsId = bsm.bookingId AND bsm.isDeleteFl = 'N' 
LEFT JOIN clientBranchMaster pcbm ON bd.pickupBranchId = pcbm.clientBranchId AND pcbm.isDeleteFl = 'N' 
LEFT JOIN clientBranchMaster dcbm ON bd.deliverBranchId = dcbm.clientBranchId AND dcbm.isDeleteFl = 'N' 
LEFT JOIN shipmentorderdetails sod ON bsm.shipmentOrderId = sod.orderId AND sod.isDeleteFl = 'N' 
LEFT JOIN BookingAdditionalDetails bad ON bad.bookingDetailsId = bd.bookingDetailsId 
LEFT JOIN shipmentdetails sd ON sod.currentMilestoneId = sd.shipmentId AND sod.isDeleteFl = 'N' 
WHERE bd.bookingDetailsId = :bookingDetailsId

(B.)Activity Logs-[t]


 Shippers can also view the ‘Activity Logs’ to see the latest updates on their order requests. The Activity Log will have the following information: 
      1. Search - On the search bar there will be a list of activities from which shippers can select the activity on which they want to perform search. Shipper can multi-select the activities and then can enter the keywords on the search bar and click on the search icon. The activity log will be refreshed based on the search parameters and in case there are no results available then an empty view with message ‘No activity logs found’ will be shown to the user. These are following list of activities that will be available:
         1. Order Status - Any update made on the status of the order. 
            1. Order request created by <shipper username>, Created on <Datetime>
            2. Order request status changed to <> from <>, Updated on <Datetime> by <dispatcher username>.
         2. Customer Updates- Pickup/Delivery/Return Address 
         3. Shipment Details - This includes Dispatch date, Payment type, Skill set, Discount, Secure Shipment.
         4. Item Details
         5. Exceptions
         6. Comments
         7. Payment/Cost related changes
      2. Date filter- There will be a date filter on the activity log section. Shipper can click on it and apply a filter. The filter will have following options : 
         1. All
         2. Today
         3. Yesterday
         4. Last 7 days
         5. Last 30 days
         6. Last 1 year
         7. Custom- Select a custom range. 
      3. The activity logs timeline will show following information:
         1. Date-Each update will be segregated into date wise sections. 
         2. Update on date time stamp
         3. Created on date time stamp
         4. Order request updates
         5. Username who updated the change
      4. If at any update there are EPOD/Esign documents attached, then the shipper can click on the update card and it will open the attached epdo/e-sign on the right side. The information under EPOD/Esign will be Receiver’s name, signature, Proof images. On clicking the proof image, it shall open the preview and the shipper can click on the carousel to view next/previous images. [u]

    sim.imageName, sim.imageType,  sim.comments,  sim.createdOnDt,
    sim.updatedOnDt,  sim.shipmentLocationId AS shipmentLocationId,
    cn.emailAddress, cn.clientNodePhone
FROM ShipmentDetails sd
inner JOIN Shipmentimagemap sim ON sim.shipmentDetailsId = sd.shipmentId AND sd.isDeleteFl = 'N'
LEFT JOIN shipmentlocations sl ON sl.shipmentLocationId = sim.shipmentLocationId AND sim.isDeleteFl = 'N'
LEFT JOIN clientNode cn ON cn.clientNodeId = sl.clientNodeId
WHERE  sd.shipmentorderid = :orderId  AND sd.movementType IN ('FMOH', 'DHLM');

API to fetch messages in OrderLevel or in Bulk : In this API I found a logic that what the response will be after pagination it will convert the final response in such a way via below piece of code.


 
Collections.sort(messageList, (a, b) -> b.getCreatedOnDate().compareTo(a.getCreatedOnDate()));

This piece of code is sorting a list of messages (messageList) in descending order based on their creation date. Here's a breakdown of what each part does:
      1. Collections.sort(messageList, ...): This is a static method from the Collections class that sorts the messageList using a custom comparator.
      2. (a, b) -> : This is a lambda expression defining the comparator. It compares two elements (a and b) from the messageList.
      3. b.getCreatedOnDate().compareTo(a.getCreatedOnDate()): This part compares the creation dates of the two elements. b.getCreatedOnDate() retrieves the creation date of message b, and a.getCreatedOnDate() retrieves the creation date of message a. The compareTo method is then used to compare these dates. By placing b before a in the comparison, the list will be sorted in descending order of creation date (i.e., newer messages will come before older ones).


      * So to avoid this for SHIPPER I need to add a shipper, dispatcher condition as discussed or confirmed by Rishabh after today’s 1:00 pm SCRUM call, dispatcher don’t have this kind of sorting but shipper will be needing it, so accordingly changed the code[dv].
      * Also did the same with the App to Service Call API where pagination was not sent to added code according to the shipper, dispatcher condition like below.
